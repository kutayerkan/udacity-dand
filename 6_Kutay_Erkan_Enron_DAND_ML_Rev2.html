Summary¶

In this project we aim to create a Machine Learning model and a POI (Person of Interest) Identifier by using Enron email and financial data. Machine Learning is an appropriate tool for this project as it is more feasible to use finely tuned features rather than simple, deterministic rules to conclude there is a fraud; and Machine Learning is much more effective than a human when scale of the problem goes from a couple of documents to hundreds.

Dataset¶

Dataset includes information from Enron employees, namely emails and financial data, which were released to public after Enron's collapse in 2001. The dataset is compiled by Udacity for the purposes of this project.

Other than financial data that we can directly use, we have also email records. We already have the number of emails sent to and received from POI's by a person, but what if that person inherently has a large number of emails?(e.g. an assistant or a secretary). Thus it is logical to engineer new features by dividing POI emails to number of all emails of that person.

  • New Feature #1: fraction_from_poi (from_poi_to_this_person/to_messages)
  • New Feature #2: fraction_to_poi (from_this_person_to_poi/from_messages)
In [9]:
# defining a function to compute new features

def computeFraction( poi_messages, all_messages ):
    
    fraction = 0
    
    if poi_messages != "NaN" and all_messages != "NaN" and all_messages != 0:
        fraction = float(poi_messages) / all_messages
    else:
        return 0

    return fraction

Having defined the function, we compute and add new features to our dataset.

In [10]:
# adding new features to data dictionary

for name in data_dict:

    data_point = data_dict[name]

    
    from_poi_to_this_person = data_point["from_poi_to_this_person"]
    to_messages = data_point["to_messages"]
    fraction_from_poi = computeFraction( from_poi_to_this_person, to_messages )
    data_point["fraction_from_poi"] = fraction_from_poi


    from_this_person_to_poi = data_point["from_this_person_to_poi"]
    from_messages = data_point["from_messages"]
    fraction_to_poi = computeFraction( from_this_person_to_poi, from_messages )
    data_point["fraction_to_poi"] = fraction_to_poi

Tuning¶

4. What does it mean to tune the parameters of an algorithm, and what can happen if you don’t do this well? How did you tune the parameters of your particular algorithm? What parameters did you tune? (Some algorithms do not have parameters that you need to tune -- if this is the case for the one you picked, identify and briefly explain how you would have done it for the model that was not your final choice or a different model that does utilize parameter tuning, e.g. a decision tree classifier). [relevant rubric items: “discuss parameter tuning”, “tune the algorithm”]

Tuning means changing and updating parameters so that our algorithms work in the most suitable and optimal way for our context. Most of the parameters use a default and we might not have to change it, but in some cases it is necessary to change the parameter. Reason behind this could be getting most correct results, such as shuffling the data in the feature selection. We might also want to change minimum split in a decision tree so that we can decrease model fitting from minutes to seconds. Or we might simply set a random state in nondeterministic algorithms, so that our results become reproducible by someone else.

Following parameter tuning has been done to algorithms(scores are computed using tester.py):

  • For the Decision Tree, changing criterion from "gini" to "entropy" increased precision to 0.41348, recall to 0.41100 and F1 score to 0.41224. Increasing minimum saple leaf from default 1 to 4 increased precision to 0.48469, decreased recall to 0.38000 and increased F1 score to 0.42601.
  • For the Gaussian Naive Bayes, changing priors to [0.95, 0.05] increased precision to 0.20686, decreased recall to 0.75050 and increased F1 score to 0.32433.
  • For Logistic Regression, different "C" values for regularization has been tried using GridSearchCV but default value 1.0 produced optimal results. "liblinear" solver also hasn't been changed due to our small dataset.

Validation¶

5. What is validation, and what’s a classic mistake you can make if you do it wrong? How did you validate your analysis? [relevant rubric items: “discuss validation”, “validation strategy”]

Validation, in general terms, testing your model with the data you haven't "touched". If a big enough training set hasn't been created, or the created test set has been used in model creation (I did this mistake in this project before correcting it) overfitting can happen. Another problem that undermines validation is wrong assumptions. For example if we assume our features are independent but they are highly correlated, it could damage model results (possibly) and model credibility (definitely).

For validation, a test set has been split using train_test_split module from scikit-learn. 20% (29 people) of the dataset has been separated as test set, low enough to feed enough data for modelling but high enough to trust the results generated using test set.

Evaluation¶

6. Give at least 2 evaluation metrics and your average performance for each of them. Explain an interpretation of your metrics that says something human-understandable about your algorithm’s performance. [relevant rubric item: “usage of evaluation metrics”]

Main evaluation metrics used in Machine Learning is as follows:

  • Accuracy: Fraction of predictions (by the model) that are correct.
  • Recall: Fraction of all true positive observations that are predicted correctly by the model.
  • Precision: Fraction of positive predictions that are correct.
  • F1 Score: Harmonic mean of Recall and Precision.
